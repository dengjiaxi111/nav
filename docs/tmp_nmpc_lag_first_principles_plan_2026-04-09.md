# NMPC-底盘时延问题第一性原则审查计划（临时）

日期：2026-04-09
范围：只做建模合理性审查与验证方案设计；不改控制代码。
协作目标：给同事一份可执行的“观测增强 + `tau_v` 辨识 + 参数回灌”说明。

## 0. 问题重述（统一术语）
- 已观测现象：从 NMPC 发出 `cmd_vel` 到底盘速度反馈接近该命令，约有 `0.5 s` 量级滞后。
- 已做操作：在 NMPC 模型中引入一阶惯性（`tau_v`, `tau_w`）。
- 当前疑问：
  1. 这个建模是否合理；
  2. `tau` 应该如何辨识与调节；
  3. 应该比较哪些信号来验证 `tau`。

## 1. 第一性原则：识别主导环节并最小化建模复杂度
任何“命令到速度”的滞后都可能由三部分叠加：
1. 纯传输/调度延迟 `T_d`：通信、线程调度、发布订阅时间戳差；
2. 底层闭环动态 `G(s)`：LQR+执行器+机械系统的低通响应（通常可近似为一阶或二阶）；
3. 测量链路动态 `H(s)`：里程计估计器滤波与输出延迟。

本项目已确认通信与链路延迟仅 ms 级，当前迭代采用最小复杂度策略：优先把主导滞后建模为等效一阶 `tau_v`，不单独测 `T_d`。仅当拟合残差表现出明显固定时移时，再回补 `T_d` 排查。

## 2. 对“加一阶惯性”合理性的初步结论
结论：**合理，但有边界条件**。

合理之处：
- 你们当前模型已把底盘响应写成
  - `dv/dt = (v_cmd - v)/tau_v`
  - `domega/dt = (omega_cmd - omega)/tau_w`
  这与“底层闭环表现为低通”现象一致，是 MPC 常见做法。
- 在未知精确电控内部模型时，一阶近似可显著降低模型失配。

边界与风险：
- 若主要问题是纯延迟 `T_d`（而非惯性），仅用一阶环节会产生系统性相位误差；
- 若响应明显二阶（超调/振荡），一阶会欠拟合；
- 若里程计本身有较大滤波延迟，辨识出的 `tau` 会偏大。

因此：一阶惯性可作为**最小可用模型**，但需通过实验确认“它解释了多少滞后”。

## 3. 你提的问题：该比较什么信号？
你的判断方向是对的，但需要更严格：

应该至少做两组对比：
1. `v_cmd`（MPC下发） vs `v_meas`（底盘/odom反馈）
   - 用于识别真实被控对象（命令到实速）的整体响应；
2. `v_pred`（MPC模型预测的速度状态） vs `v_meas`
   - 用于验证“当前 `tau` 是否让内部模型与现实一致”。

关键点：
- 仅比较 `v_cmd` vs `v_meas` 只能说明系统有滞后，不能直接证明 MPC 内部模型正确；
- 仅比较 `v_pred` vs `v_meas` 也不够，因为 `v_pred` 可能在错误输入/时间对齐下“看起来正确”。

所以需要“两条证据链同时成立”。

## 4. 验证流程（先测量，再辨识，再回灌）

### Step 1: 时间对齐与时钟一致性核查
- 检查 NUC 与底盘时间基准（是否已有同步机制）。
- 记录各信号时间戳来源（ROS header stamp / 接收时刻）。
- 目标：确保辨识与对比使用统一时间基准；本轮不单独估计纯延迟 `T_d`。

### Step 2: 开环辨识实验（低风险）
- 在安全场地，绕过路径跟踪干扰，做速度激励：
  - 线速度：阶跃、双阶跃、PRBS（三选二至少）；
  - 角速度：同理单独测试。
- 记录：`cmd_vel`, `odom.twist`,（若可得）底盘内部速度估计。
- 提取：上升时间、63% 时间、稳态误差、等效纯延迟。

### Step 3: 建模结构选择
- 先拟合等效一阶：`G(s)=K/(tau s+1)`（本轮默认 `K≈1`）。
- 判据：
  - 若残差小且跨工况稳定，保留一阶；
  - 若存在明显超调或双时间常数，升级为二阶/串联一阶；
  - 若存在稳定固定时移（相位整体平移），再将 `T_d` 作为可选项引入：`K*exp(-T_d s)/(tau s+1)`。

### Step 4: 将辨识结果映射到 MPC 参数
- 将 `tau_v/tau_w` 设置为辨识值（先保守，后微调）。
- 明确区分：
  - `tau` 负责动态惯性；
  - 前馈项（如 `vel_ff_time`）负责相位补偿；
  - `odom_feedback_alpha` 影响状态估计混合，不应与 `tau` 混调。

### Step 5: 闭环验收（路径场景）
- 使用同一路径、同速档、同负载做 A/B：
  - A: 原参数；B: 新 `tau`。
- 评价指标：
  - 轨迹误差（横向/航向）；
  - 速度跟踪误差 RMSE；
  - 控制平滑性（加速度/角加速度抖动）；
  - 到点时间与稳定性。

## 8. 已确认前提（2026-04-09）
1. 约 `0.5 s` 滞后主要来自斜坡激励下的响应表现（非单纯路径跟踪主观观感）。
2. 通信延迟量级仅数 ms，可视为非主导项。
3. 可获取嵌入式内部速度估计并发布到 ROS；HS USB CDC 链路延迟可认为非主导项。
4. 本轮辨识只做线速度通道：`tau_v`；角速度延迟仅几十 ms，暂不作为补偿重点。
5. 测试频率：控制 50 Hz，反馈 50 Hz。
6. 对照信号约定：
   - 命令：`/cmd_vel`
   - 实测：`/ChassisOdom`

据此，本问题当前优先建模为：`动态低通惯性主导`，辨识重点放在 `tau_v`；`T_d` 为可选排查项（默认不测）。

## 9. 关键设计共识（必须遵守）
1. 不以“`cmd_vel` 数学阶跃”作为目标；目标是“约束内最快可实现响应 + 可预测性”。
2. 不取消 MPC 加速度约束（`a_cmd` 约束）：
   - 原因：若去掉约束，优化器会产生不可执行的尖峰 `a_cmd`，内环再饱和后将放大模型失配。
   - 做法：可在验证后“逐步放宽”，不能直接“无约束”。
3. 内环 LQR 的限幅能力不能替代外环可执行性约束；内环限幅应被外环视为对象特性的一部分。
4. 先保证模型匹配（`v_pred` 对齐 `v_meas`），再追求激进响应（调 `R_linear`、`max_linear_accel`）。

3. 任何参数回灌必须保留前一版快照，可一键回滚。

---
临时结论（本阶段）：
- 在你给出的背景下，引入一阶惯性建模是合理的最小方案；
- `tau` 不应靠主观手调，必须先做“纯延迟 vs 动态惯性”分离辨识；
- 验证 `tau` 时，**要对比 MPC 预测速度与实测速度**，但要与 `cmd_vel->实测` 证据链配套。
