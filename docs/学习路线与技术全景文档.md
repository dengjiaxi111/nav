## 2. 系统技术文档（深入浅出）

## 2.1 一句话理解系统

这是一个 ROS2 导航系统：  
**LiDAR-IMU 里程计 + NDT 全局对齐 + 3D/2D 地图融合 + A*/B样条路径 + NMPC 控制 + 恢复/脱困机制**。

---

## 2.2 模块架构与职责

### A. 定位层
- `small_point_lio`
  - 输入：激光 + IMU
  - 输出：`/Odometry`、`/cloud_registered`
  - 本质：高频局部里程计（`odom->base_link`）

### B. 全局重定位层
- `localization_initializer`
  - 输入：`/cloud_registered` + RViz `/initialpose` + PCD 地图
  - 输出：`map->odom` TF
  - 本质：把局部里程计锚定到全局地图

### C. 地图层
- `rog_map_ros2_node`（integration）
  - 输入：点云、里程计
  - 输出：`/rog_map/map_2d`（局部动态栅格）
- `LayeredMapManager`
  - 静态地图 + 动态层融合，生成：
  - `static_map` / `fused_map` / `costmap` / `esdf_map`

### D. 规划控制层
- `SimplePlanner`
  - A* 搜索 + B 样条平滑 + ESDF 约束优化
- `NMPC`
  - 基于 acados 的非线性 MPC，输出 `cmd_vel`
- `NavServer`
  - 状态机总控：接目标、规划、控制、恢复、脱困、发布反馈

### E. 任务决策与串口层（扩展）
- `sentry_decision`
  - 发布 `/sentry/target_position`
- `myserial`
  - 与下位机通讯，转发裁判系统和控制状态

## 2.3 端到端数据流（必须会背）

1. 传感器：LiDAR/IMU -> `small_point_lio`
2. `small_point_lio` 输出 `/Odometry` 与 `/cloud_registered`
3. `localization_initializer` 结合先验地图做 NDT，对齐后发布 `map->odom`
4. `rog_map` 维护 3D 地图并投影成 `/rog_map/map_2d`
5. `NavServer` 的 `LayeredMapManager` 融合静态图和动态层，生成 `costmap/esdf`
6. `SimplePlanner` 基于 `costmap` 规划路径并平滑优化
7. `NMPC` 根据路径、里程计、ESDF 求解控制，发布 `/cmd_vel`
8. 如遇异常，状态机转入 `ESCAPING/RECOVERY` 再回到 `PLANNING`

## 2.4 数学原理（按代码实现）

## 2.4.1 A*（栅格搜索）
- 代价：`f(n)=g(n)+h(n)`
- `g`：走过路径长度 + 栅格代价加权
- `h`：欧式距离启发
- 障碍判定来自 `costmap`，阈值由 `planner.obstacle_threshold`

## 2.4.2 B 样条平滑与优化
- 先对 A* 路径做等弧长重采样
- 三次 B 样条参数化
- 优化目标（核心）：
  - 平滑项：离散二阶差分（抑制折线抖动）
  - 碰撞项：`(d_esdf - safe_dist)^2`（只在距离不足时生效）
  - 可选：窄通道中心吸引、台阶方向约束

可记为：
\[
J = \lambda_s J_{smooth} + \lambda_c J_{collision} + \lambda_a J_{align} + \lambda_t J_{stair}
\]

## 2.4.3 ESDF（欧氏距离场）
- 每个栅格存“到最近障碍物的距离”
- 作用：
  - 规划平滑时给“远离障碍物”的连续梯度
  - NMPC 中给软约束代价

## 2.4.4 NMPC（模型预测控制）

状态（7 维）：
\[
x=[p_x,p_y,\theta,v,\omega,v_{cmd},\omega_{cmd}]
\]

控制（2 维）：
\[
u=[a_{cmd},\alpha_{cmd}]
\]

运动学/动力学（含一阶滞后）：
\[
\dot p_x=v\cos\theta,\quad \dot p_y=v\sin\theta,\quad \dot\theta=\omega
\]
\[
\dot v = \frac{v_{cmd}-v}{\tau_v},\quad \dot \omega = \frac{\omega_{cmd}-\omega}{\tau_\omega}
\]
\[
\dot v_{cmd}=a_{cmd},\quad \dot\omega_{cmd}=\alpha_{cmd}
\]

代价核心：
- 跟踪误差（位置/航向/速度）
- 控制平滑（加速度惩罚）
- ESDF 软约束（离障碍太近会罚）
- Contouring（抑制“切西瓜”偏离参考线）

## 2.5 NavServer 状态机与工程机制

状态：`IDLE / PLANNING / CONTROLLING / ESCAPING / RECOVERY / SUCCEEDED / FAILED`

关键机制：
- 多目标入口：Action `navigate` 与 RViz `goal_pose`
- 路径周期检查：路径被动态障碍阻断或偏航过大时重规划
- 起点占障碍触发脱困（先找安全点，再转向+直行）
- 恢复行为链：后退、自转等
- 台阶模式 `stair_mode`：靠近台阶时发布模式给底盘控制策略联动

## 2.6 关键参数怎么理解（按目标拆）

## 2.6.1 想“更稳，不撞墙”
- `inflation.radius` 增大
- `planner.opt_lambda_collision` 增大
- `planner.opt_safe_dist` 增大
- `nmpc.esdf_safe_dist` 增大
- `nmpc.esdf_weight` 增大

## 2.6.2 想“更快”
- `nmpc.max_linear_vel` 增大
- `nmpc.desired_velocity` 增大
- `nmpc.max_linear_accel` 适度增大
- 同时注意 `Q/R` 平衡，避免高频震荡

## 2.6.3 想“弯道不甩尾/不抖”
- `nmpc.Q_orientation`、`nmpc.Q_omega` 适度提高
- `nmpc.R_angular` 适度提高（抑制角加速度）
- `nmpc.enable_curvature_horizon_adapt=true`

## 2.6.4 想“窄通道居中”
- `planner.narrow_passage_align=true`
- 调 `planner.narrow_passage_thresh` 与 `planner.opt_lambda_align`

## 2.6.5 想“过台阶更稳”
- `special_terrain.enable_stair_layer=true`
- `special_terrain.enable_stair_mode_detection=true`
- 根据底盘特性调 `stair_mode_*` 与 `stair_fixed_*`

## 2.8 常见 Bug 排查（现象 -> 检查 -> 处理）

## 1. 无法规划 / 一直“等待地图”
- 检查：
  - `map_file` 是否有效
  - `static_map/fused_map/costmap` 是否发布
- 处理：
  - 先禁用动态层只跑静态层，确认基础可规划后再叠加动态层

## 2. 机器人位置漂移、路径“平移错位”
- 检查：
  - `map->odom->base_link` TF 是否连续
  - `localization_initializer` 是否完成初始化
- 处理：
  - 重新给 `2D Pose Estimate`
  - 修正 `base_link<->livox_frame` 外参

## 3. 路径贴墙或切角过猛
- 检查：
  - `inflation` 是否过小
  - `opt_safe_dist`、`esdf_weight` 是否过低
- 处理：
  - 先增大地图安全边界，再增加优化避障权重

## 4. 控制抖动、角速度尖峰
- 检查：
  - `Q_omega / R_angular` 配比
  - `vel_lag_tau / omega_lag_tau` 与底盘响应是否匹配
- 处理：
  - 增大 `R_angular`，适度增大 `omega_lag_tau`，必要时降 `desired_velocity`

## 5. 起点在障碍物中原地打转
- 检查：
  - `escape.trigger_costmap_threshold` 与 `escape.target_safe_esdf`
  - `costmap` 是否膨胀过重导致可行区被吃掉
- 处理：
  - 适度降低触发阈值或放宽脱困目标，必要时减小膨胀半径
